## 2. BubbleTea Patterns

### 2a. Model Structure / Elm Architecture

**Root model**: `Model` is a large flat struct (~200 fields) in `root.go`. Value receiver on `Update()` and `View()` (copies returned), pointer receiver on mutation helpers (`dispatchAction`, `layout`, `persistPermissionRule`, etc.). This is a **mixed receiver pattern** — risky. Mutation via pointer-receiver helpers called from value-receiver `Update()` works only because the pointer-receiver methods operate on `m` before it's returned, but it's fragile and non-obvious.

**Child components**: Most overlays (`agentselector.Model`, `modelselector.Model`, `commandpalette.Model`, etc.) follow Elm properly — value receivers on `Update()/View()`, return new model copy. They implement no `tea.Model` interface; root drives them inline.

**Panel interface** (`panels/panel.go:23`): Uses a **custom non-Elm interface**:
```go
Update(msg tea.KeyMsg) (tea.Cmd, bool)  // bool = "consumed"
View() string
```
Panels are NOT `tea.Model`. They receive only `tea.KeyMsg`, not the full msg stream. State mutations happen via pointer receivers on panel structs (they are held as pointers in root). This is intentional and pragmatic, but breaks pure Elm — panels can't receive non-key messages or emit arbitrary Cmds.

### 2b. Side Effects in Update/View

**Violations found:**

#### Direct I/O in `Update()`:
- `root.go:1341` — `ReadImageFile(absPath)` (disk read) inside `case filepicker.SelectMsg:` in `Update()`
- `root.go:1471` — `os.ReadFile(msg.TmpPath)` + `os.Remove(msg.TmpPath)` inside `case memorypanel.NewMemoryMsg:` in `Update()`
- `root.go:1554` — `os.ReadFile(msg.path)` inside `case editorDoneMsg:` in `Update()`
- `root.go:3668,3678` — `persistPermissionRule()` (calls `os.ReadFile` + `os.Stat` + file write) inside `handlePermissionResponse()` which is called from `Update()` — blocks on disk I/O during permission approval
- `root.go:3664,3669,3679,3682` — `m.approvalCh <- true/false` (channel send) inside `Update()` — **potentially blocking** if tool goroutine isn't reading yet; channel is buffered(1) so mostly safe but still a side effect in Update

#### Direct I/O in `View()`:
- `root.go:3613` — `os.ReadFile(m.planFilePath)` inside `renderPlanApprovalDialog()` which is called from `View()` — **disk read on every render frame** while plan approval dialog is visible

### 2c. WindowSizeMsg Propagation

- **Root** (`root.go:712`): Handles `tea.WindowSizeMsg`, updates `m.width/m.height`, calls `m.layout()` + `m.refreshViewport()`. Returns `nil` cmd (no child propagation).
- **Child components**: None receive `WindowSizeMsg` directly via BubbleTea's normal msg dispatch. Root calls `SetSize(w, h)` or `SetWidth(w)` imperatively when it knows dimensions. This is intentional (root owns layout) but means child components can't react to resize autonomously.
- **Panels**: Sized via `panel.SetSize(w, h)` called from `renderPanelWithHelp()` inside `View()` — sizing happens at render time, not on resize event. Works correctly but panels don't get an opportunity to recompute layout state on resize (must recompute every View call).
- **Selectors/overlays**: `SetWidth`/`SetHeight` called imperatively in `WindowSizeMsg` case (`root.go:718-720`).

### 2d. Direct I/O in Update/View

Summarized above. Key violations:
- 3× `os.ReadFile` in Update (lines 1341, 1471, 1554)
- 1× `os.ReadFile` + `os.Stat` + `os.WriteFile` in Update via `persistPermissionRule` (line 3668/3678)
- 1× `os.ReadFile` in View (line 3613) — worst offender: runs every frame

## 3. Key Handling

### BubbleTea Version
**v1.3.10 (v1, NOT v2)**. Uses `tea.KeyMsg` throughout — correct for v1. `tea.KeyPressMsg` does not exist in this version.

### Keymap Architecture
**Centralized and well-structured:**
- `keymap/actions.go` — `ActionID` iota enum for all actions
- `keymap/keymap.go` — `Keymap` struct with `Resolve(seq string) ActionID` and `HasPrefix(seq string) bool` for leader-key disambiguation
- `root.go:3825` — `handleLeaderKey()` implements vim-style leader sequence with timeout (WhichKey popup scheduled via `whichkey.ScheduleTimeout`)
- `root.go:3878` — `dispatchAction(ActionID)` — single dispatch point for all bound actions

**Inline key checks (scattered)** also exist in `Update()` for context-specific keys:
- `root.go:1207-1241` — vim Normal mode viewport scrolling (j/k/ctrl+d/ctrl+u/G/g)
- `root.go:1244-1302` — focus-routing to overlays when active (model/agent/team selector, permission dialog, etc.)
- `agentselector/selector.go:106` — `key.String()` switch directly in selector Update
- `teamselector/selector.go:101` — same pattern
- Panel `Update(tea.KeyMsg)` — each panel has its own switch/if chain

**Pattern**: Leader keys → centralized dispatch. Modal keys (overlays, dialogs) → inline in Update per-context. This is reasonable but the root `Update()` switch on `tea.KeyMsg` is long.

## 5. Component Composition

### Init() delegation
`root.go:686` — `Model.Init()` returns `tea.Batch(tea.SetWindowTitle, tea.EnableBracketedPaste, waitForEvent(), logoTick, taskTick)`. **No child `Init()` calls** — none of the sub-models implement `tea.Model.Init()` since they're not `tea.Model` implementations.

### Child Cmd batching
Properly done where applicable:
- `root.go:1246-1302` — overlay updates accumulate to `cmds []tea.Cmd`, returned as `tea.Batch(cmds...)`
- `root.go:2374` — `tea.Batch(m.spinner.Tick(), m.waitForEvent(), timerTick)` correctly batches 3 cmds
- 16 `tea.Batch` callsites in root.go total — consistent pattern

### Panel integration
Panels hold state internally (pointer in root struct). Root calls `panel.Update(tea.KeyMsg)` and collects returned `tea.Cmd`. Panels emit `panels.ActionMsg` or `panels.CloseMsg` via... wait, they don't — panels' `Update` returns `(tea.Cmd, bool)`. Any message-passing from panels to root must go through `tea.Cmd` that produces a recognized message. This works but panels have no way to emit arbitrary messages back to root without tea Cmd wrapping.

## 8. State Management

### Shared state mechanisms:

1. **AppContext** (`context.go`) — fat service bag injected into root at construction via `WithAppContext()`. Root passes sub-services to panels imperatively (e.g., `tasksPanel.SetRuntime(ctx.TaskRuntime)`). No reactive propagation.

2. **eventCh** (`chan tuiEvent`, buffer 64) — primary async bridge from engine goroutine → BubbleTea loop. `waitForEvent()` Cmd blocks on channel receive, re-schedules itself on each message. Classic long-poll pattern for BubbleTea.

3. **Event Bus** (`bus.Bus`) — `root.go:379` subscribes to `EventBgTaskComplete` from a background goroutine callback. The callback pushes to `eventCh` with a non-blocking select (drops if full). Goroutine lifecycle: bus subscription has no unsubscribe — goroutine runs for program lifetime.

4. **SessionRuntime** (`sessionrt.go`) — encapsulates per-session state (messages, streaming state, tokens/cost). Background drain goroutine (`drainLoop`) consumes engine events when session is backgrounded. Properly started/stopped via `StartBackgroundDrain`/`StopBackgroundDrain` with `sync.Mutex` and `drainStop` channel.

5. **Direct struct access** — panels receive references to AppContext services and read/write them directly. No message-passing protocol between panels and root (aside from tea.Cmd returns).

6. **AskUser channels** (`reqCh`/`respCh`) — `root.go:2299-2309` wires tool channels and spawns a goroutine to forward requests to eventCh. Goroutine runs until `reqCh` is closed; no explicit cleanup hook.

## 9. Error Handling

### Engine errors:
- `root.go:2562-2563` — `event.err` from engine "done" event: added as `MsgError` chat message. Ignores "context canceled" (user-cancelled). Visible in chat.

### Cmd errors:
- No global error handler. Each Update case handles its own errors inline (usually `m.addMessage(ChatMessage{Type: MsgError, ...})`).

### I/O errors in Update:
- `root.go:1471` — `os.ReadFile` error silently drops the new memory entry (no user feedback)
- `root.go:1554` — `os.ReadFile` error silently ignores prompt restoration
- `root.go:3790` — `os.ReadFile` error in `persistPermissionRule` silently continues with empty rules list — permission rule may be lost

### I/O errors in View:
- `root.go:3613` — `os.ReadFile` error silently skips plan preview — no user feedback, plan approval dialog shows without content

## 10. Anti-Patterns & Risks

### 🔴 Critical

1. **`os.ReadFile` in `View()`** (`root.go:3613` in `renderPlanApprovalDialog`): Disk I/O on every render frame during plan approval. Will stutter/block on slow filesystems. Fix: cache plan content in model state when `FocusPlanApproval` is set in Update.

2. **Mixed value/pointer receivers on root Model**: `Update()` is value receiver, but calls pointer-receiver helpers (`m.layout()`, `m.persistPermissionRule()`, `m.dispatchAction()`). The pointer methods mutate the copy on stack, not the original. This only works correctly because each call is: take `m` (copy), call pointer method on `&m` (mutates copy), return `m`. If any path returns before mutations complete, state is lost silently. Very subtle bug surface.

3. **`persistPermissionRule()` called in `Update()`** (`root.go:3668,3678`): Synchronous `os.ReadFile` + `os.Stat` + `json.Marshal` + `os.WriteFile` chain during permission approval. Blocks BubbleTea event loop. Fix: move to tea.Cmd.

### 🟡 Moderate

4. **`ReadImageFile()` in `Update()`** (`root.go:1341`): Synchronous disk read (potentially large image) during file picker selection message handling. Fix: move to tea.Cmd.

5. **Bus subscription goroutine without cleanup** (`root.go:379`): `Bus.Subscribe` callback closure runs forever; no unsubscribe on model teardown. Acceptable for single-session lifetime but leaks if Model is ever reconstructed.

6. **AskUser forwarding goroutine** (`root.go:2305-2309`): Goroutine reads `reqCh` until closed. If `reqCh` is never closed (e.g. tool panics), goroutine leaks. No timeout or context cancellation.

7. **Sidebar rebuilt every View frame** (`root.go:6388`): `m.sidebar = m.buildSidebar()` called inside `View()` on every render. Comment explains it's needed to capture current value-receiver closure. Root cause is the value-receiver/closure capture problem — fix at Model design level would eliminate this.

8. **Hardcoded colors in `notifications/queue.go`** (lines 221-241): Entire notification card color system bypasses theme. Dark-mode/light-mode switching would require touching this file.

9. **Hardcoded `#e78a1f` in `panels/agui/panel.go:691-692`**: Tool running status color not in theme.

### 🟢 Minor / Observations

10. **No `Init()` chain**: Child components have no `Init()` because they're not `tea.Model`. Fine for v1 pattern but means no child can schedule startup Cmds.

11. **Panel interface receives only `tea.KeyMsg`**: Panels cannot react to `tea.WindowSizeMsg`, custom messages, or ticks directly. They're sized imperatively by root. Works but limits panel autonomy.

12. **`layout()` called from `View()`** (`root.go:6296`): View mutates `m.viewport.Width/Height` via pointer-receiver `layout()`. This means viewport dimensions change during rendering. Works due to value-receiver copy semantics (View operates on its own copy) but is conceptually wrong — layout should be finalized in Update on `WindowSizeMsg`, not during View.

13. **`m.sessionPicker.SetSize()` in `View()`** (`root.go:6331`): Imperative size mutation during rendering.

14. **`panel.SetSize()` in `renderPanelWithHelp()`** (`layout.go:83`): Called from View. Same pattern — sizing during render instead of in Update.

## Dependencies & Data Flow

```
Engine goroutine
      │ (engine.Run / RunWithBlocks)
      │
      ▼
  eventCh (chan tuiEvent, buf=64)
      │
      ▼
  waitForEvent() Cmd ──► BubbleTea loop
      │
      ▼
  Update(engineEventMsg) ──► handleEngineEvent()
      │                            │
      │                            ├─ text_delta → streamText builder → refreshViewport
      │                            ├─ tool_start/end → ChatMessage append
      │                            ├─ approval_needed → FocusPermission, waitForEvent
      │                            ├─ done → finalize, FocusPrompt, re-queue next msg
      │                            └─ teammate_event → agui panel, optional handleSubmit
      │
      ├─ tea.KeyMsg ──► focus-routing ──► overlay.Update OR panel.Update OR leaderKey
      │
      ├─ tea.WindowSizeMsg ──► m.width/height, m.layout(), m.refreshViewport
      │
      └─ custom Msgs (SelectMsg, SubmitMsg, etc.) ──► inline handlers
```

State shared via:
- `AppContext` (injected at construction, read-only after init)
- `eventCh` (goroutine → BubbleTea, one direction)
- `approvalCh` (BubbleTea Update → tool goroutine, one direction, buf=1)
- `bus.Bus` (broadcast, goroutine callback → eventCh)
- Panel pointer fields (root holds `*Panel`, root calls methods directly)

## Open Questions

1. **Bus unsubscribe**: Does `bus.Bus` support unsubscribing? If yes, should be called on model teardown. If no, goroutine leak is permanent but bounded by process lifetime.

2. **Value-receiver `Update` + pointer-receiver helpers**: Is the mutation-before-return pattern guaranteed safe under all code paths? Any path that returns early (via nested helper) before a pointer-mutation completes would silently lose state.

3. **`m.layout()` + `m.viewport` mutation in `View()`**: Under BubbleTea's internal model, View is supposed to be pure. Does any concurrent access (e.g. terminal resize event racing with View render) risk a torn read of `m.viewport.Width`? Likely safe in single-threaded BubbleTea event loop but worth auditing.

4. **`os.ReadFile` in `renderPlanApprovalDialog` (View)**: If the plan file is on a network filesystem or slow SSD, does this block the render thread? What's the observed latency in practice?

5. **AskUser `reqCh` lifetime**: Who closes `reqCh`? If the tool is garbage-collected without closing the channel, the forwarding goroutine at `root.go:2305` leaks indefinitely.
